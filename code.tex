\chapter{Methodology}

This chapter describes the design and implementation of the real-time multi-road adaptive traffic control system deployed at \textbf{Prithivi Chowk}, the highest-traffic intersection in Pokhara, Nepal. The system integrates computer vision, parallel video processing, adaptive signal timing, and a PyGame simulation to dynamically manage traffic at this four-way junction, optimized for real-time operation on commodity hardware.

% ----------------------------------------
\section{Product Development Lifecycle}

The development followed an iterative Agile model. Core objectives were defined as real-time parallel video processing for four approach roads, YOLO-based vehicle detection and tracking, dynamic Green Signal Time (GST) calculation using lane-specific configuration, and a PyGame-based intersection simulation.

A product backlog was created containing features such as video capture, vehicle counting, centroid-based tracking, GST computation, signal controller logic, data logging, and simulation visualization. Development proceeded in short sprints, with each module tested and integrated incrementally.

Continuous integration and iterative refinement ensured the system achieved low latency, high detection accuracy, and realistic traffic behaviour representative of Prithivi Chowk.

\section{Technologies and Development Environment}

\subsection{Computer Vision and Object Detection}
\begin{itemize}
    \item \textbf{Ultralytics YOLOv8s:} Real-time object detection model used for identifying vehicles. It detects five classes: car, motorcycle, bus, truck, and bicycle.
    \item \textbf{OpenCV (cv2):} Used for video capture, frame processing, resizing, and annotation (bounding boxes and overlays).
    \item \textbf{CentroidTracker:} Custom multi-object tracking system using the Hungarian algorithm for centroid association with direction-based filtering (right-to-left movement only).
\end{itemize}

\subsection{Signal Timing and GST Calculation}
\begin{itemize}
    \item \textbf{GST Engine:} A rule-based adaptive model that computes Green Signal Time using weighted vehicle counts and lane configurations.
    \item \textbf{Thread-safe shared state:} Implemented using \texttt{threading.Lock} to enable safe concurrent access across multiple video processing threads.
\end{itemize}

\subsection{Simulation and Visualization}
\begin{itemize}
    \item \textbf{PyGame:} Used to simulate a realistic multi-lane intersection with vehicle physics, path following, and a HUD interface.
    \item \textbf{OpenCV HighGUI:} Displays a 2$\times$2 verification grid of live video feeds with detection overlays.
\end{itemize}

\subsection{Data Output}
\begin{itemize}
    \item \textbf{CSV module:} Stores structured traffic logs and signal transitions.
    \item \textbf{JSON:} Stores GST cache and system state snapshots.
    \item \textbf{TXT reports:} Human-readable summaries of system performance and traffic statistics.
\end{itemize}

\subsection{Development Environment}
\begin{itemize}
    \item \textbf{Python 3.11}
    \item \textbf{PyTorch 2.x with CUDA acceleration}
    \item \textbf{Windows 11}
    \item \textbf{VS Code}
\end{itemize}

\section{Dataset Collection and Preparation}

The system uses four video streams representing each approach at Prithivi Chowk:

\begin{itemize}
    \item China Pull Road (\texttt{chinapull\_road.MOV}) — 6 lanes, GST: 15–44 s, straight ratio: 0.72
    \item Airport Road (\texttt{airport\_road.MOV}) — 4 lanes, GST: 15–28 s, straight ratio: 0.72
    \item Sabhagriha Chowk Road (\texttt{sabhagiriha\_road.MOV}) — 4 lanes, GST: 12–44 s, straight ratio: 0.78
    \item Naya Bazar Road (\texttt{nayabazar\_road.mp4}) — 4 lanes, GST: 20–27 s, straight ratio: 0.6
\end{itemize}

YOLOv8s is used with pre-trained COCO weights, eliminating the need for dataset-specific training.

% ----------------------------------------
\section{Data Preprocessing}

\subsection{Frame Processing}
Frames are extracted using OpenCV. Every second frame is processed (frame skip = 2), and resized to 416$\times$416 for inference.

\subsection{Detection Filtering}
Overlapping detections are merged using an IoU threshold of 0.4:

\[
\text{IoU} = \frac{\text{Intersection Area}}{\text{Union Area}}
\]

This reduces duplicate detections of the same vehicle.

\subsection{Counting Strategy}
A virtual converging count line is used. A vehicle is counted only when:
\begin{itemize}
    \item It crosses the count line,
    \item Movement direction ratio $> 0.6$,
    \item Track age $\geq 8$ frames.
\end{itemize}

\section{Feature Engineering}

\subsection{Detection Features}
\begin{itemize}
    \item Bounding box coordinates
    \item Centroid position
    \item Class label
    \item Confidence score (threshold: 0.35)
\end{itemize}

\subsection{Tracking Features}
\begin{itemize}
    \item Centroid history (20 frames)
    \item Majority-vote class correction
    \item Frame persistence tracking
    \item Minimum displacement threshold (40 px)
    \item Direction ratio filtering (> 0.6)
\end{itemize}

\subsection{GST Features}
\begin{itemize}
    \item Weighted vehicle counts
    \item Lane configuration per road
    \item Vehicle-type crossing times
    \item Min/max GST constraints
    \item Per-road straight-going vehicle ratio (replaces universal 2/3)
\end{itemize}

The GST is computed using:

\[
\text{GST} = \max\left(\text{min\_gst},\ \min\left(\text{max\_gst},\ \frac{\text{straight\_ratio} \times \sum (\text{count}_i \times \text{crossing\_time}_i)}{\text{num\_lanes} + 1}\right)\right)
\]

where \texttt{straight\_ratio} is a per-road parameter representing the proportion of vehicles going straight through the intersection.

\section{Model Selection}

\subsection{YOLOv8s}
Selected for its balance of speed and accuracy. It runs on GPU with FP16 precision and supports real-time inference.

\subsection{Centroid Tracker}
Uses Euclidean distance matching with a maximum threshold of 120 pixels and a disappearance limit of 40 frames.

\begin{figure}[H]

    \caption{Parallel video processing pipeline for each road: frame capture, YOLO inference, detection merging, centroid tracking, and count update.}

The pipeline consists of:
\begin{enumerate}
    \item \textbf{Video Processing Layer:} Four parallel threads perform YOLO detection and centroid tracking.
    \item \textbf{Shared State:} Thread-safe container storing counts, signal states, and GST values.
    \item \textbf{Signal Controller:} Computes GST dynamically before each phase switch.
    \item \textbf{Simulation Layer:} PyGame-based intersection visualization (see Figure~\ref{fig:main_arch}).
    \item \textbf{Verification Layer:} OpenCV-based 2$\times$2 live feed display.
    \item \textbf{Output Layer:} Logs all traffic and signal events.
\end{enumerate}

\section{Threading and Concurrency}

The system uses four video threads, one signal controller thread, one output logger thread, and a main simulation thread.

A producer-consumer model is used:
\begin{itemize}
    \item Video threads produce vehicle counts
    \item Signal controller consumes counts to compute GST
    \item Simulation consumes shared state for visualization
\end{itemize}

Thread safety is ensured using locks for both shared state and YOLO inference.

% ----------------------------------------
\section{System Workflow}

\begin{enumerate}
    \item System initializes and starts four video threads.
    \item First phase: China Pull Road is set to GREEN.
    \item Remaining roads accumulate vehicles simultaneously.
    \item When phase ends, GST is computed for next road using per-road straight ratio.
    \item Signal cycles continuously through all four roads; each full cycle (all four roads) increments a cycle counter visible in the simulation HUD.
    \item Simulation and verification windows update in real time.
    \item Logs are written at each signal transition.
\end{enumerate}

\section{Real-Time Detection and Optimization}

\begin{itemize}
    \item Frame skipping reduces computational load.
    \item YOLO inference is GPU-accelerated with FP16 precision.
    \item Independent trackers prevent cross-road interference.
    \item Noise is reduced using movement, direction, and persistence filters.
\end{itemize}
